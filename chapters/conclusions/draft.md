
A central finding of this thesis concerns how identity-location coupling became embedded in the transport architecture. IP addresses primarily describe network location, but TCP also uses them as part of the information that identifies an established connection. This dependence was practical under the assumption of relatively stable endpoint addresses and became an ordinary part of transport operation. It was not introduced as an explicit mechanism for binding communication identity to network location. Over time, the same information used to locate an endpoint also became part of the identity of its transport connection. This assumption became conventional enough to remain largely invisible. Mobility and multihoming expose the consequence: when the locator changes, the transport connection can fail even though the communicating applications and their session have not changed.

QUIC makes this architectural issue more visible because it provides an explicit connection identifier outside the current address pair. The address can change while the identifier continues to associate packets with the established connection state. Path validation deals with the new location separately. QUIC does more than add a mobility feature. Its migration mechanism separates two roles that traditional transport had allowed the address to perform together.

The exploratory evaluation gave a practical comparison of this difference. Plain TCP lost the original connection after the forced path or address change. The best-case TCP reconnect restored communication through a new connection to an address that was already known to the application. QUIC kept the existing connection when migration succeeded. No noticeable difference in recovery time was observed between the best-case TCP reconnect and QUIC migration in the preliminary attempts.

No clear speed advantage appeared under the tested conditions. The main benefit of QUIC was preservation of transport continuity. The optimized TCP case could restore communication quickly, but the original transport connection had already been lost. QUIC could absorb the path change within the existing connection.

QUIC does not remove identity-location coupling completely. The current address still determines where packets are delivered, path validation is required after a change, and deployment infrastructure can affect whether migration works in practice. Base QUIC also moves a connection between paths; concurrent application-data transfer over several paths requires multipath extensions.

Identity-location coupling gives QUIC connection migration its broader architectural significance. Network location remains necessary for packet delivery, but a change of location no longer has to terminate the transport connection. The connection can remain intact while the path to its peer changes.

Future work could extend the exploratory evaluation with repeated measurements under controlled delay, loss, and bandwidth conditions. NAT rebinding and load-balanced deployments would provide additional forms of address change. Multipath QUIC could extend the study from connection continuity after a path change to simultaneous use of several paths.
