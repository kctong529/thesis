
This thesis analyzed the extent to which QUIC connection migration mitigates the identity-location problem at the transport layer. The starting point was the observation that traditional transport protocols, especially TCP, associate connection state with IP addresses and port numbers. This design is simple and efficient when endpoint addresses remain stable, but it becomes fragile when a host changes network attachment, uses multiple interfaces, or experiences address changes caused by middleboxes.

The literature-based analysis showed that mobility and multihoming expose a long-standing architectural tension in the Internet. Communication state is maintained at the endpoints, while the network provides best-effort packet delivery. Since the network does not track sessions, endpoints must associate arriving packets with stored transport state using information carried in packet headers. In the classical model, address information therefore acts both as a locator and as part of the connection identity.

Earlier approaches addressed this problem at different layers. Mobile IP introduced network-layer indirection, HIP introduced a separate identity namespace, SCTP and MPTCP extended transport behavior, and application-layer approaches worked around limitations in the transport abstraction. These approaches demonstrate that the problem has been recognized for decades, but they also show the difficulty of deploying architectural changes across the open Internet.

QUIC provides a different transport-layer response. By introducing explicit connection identifiers, QUIC separates connection identity from the current address pair. Path validation then allows the protocol to test whether a new path can be used safely before sustained communication continues. Together, these mechanisms allow QUIC to treat address change as a transport event inside an existing connection rather than as an immediate connection failure.

The exploratory evaluation supported this distinction by comparing three cases under the same forced path or address change. Plain TCP did not provide connection continuity, since the original connection remained tied to the previous address pair. TCP with explicit reconnect restored communication at the application level, but only by creating a new transport connection. QUIC preserved transport-level continuity when migration completed successfully. The preliminary timing observations did not show a clear recovery-time advantage for QUIC over the best-case TCP reconnect baseline. Therefore, the main difference was qualitative: QUIC can preserve the existing connection context, while TCP reconnect restores communication after transport failure.

However, the evaluation also shows that QUIC should not be interpreted as a complete solution to mobility and multihoming. QUIC migration still depends on implementation support, endpoint policy, path validation, congestion-control behavior, and network conditions. In addition, base QUIC supports migration between paths, but general simultaneous use of multiple paths belongs to extension work rather than the core protocol. QUIC therefore mitigates the identity-location problem at the transport layer, but it does not remove the broader architectural dependence on network locators.

The main conclusion is that QUIC connection migration provides a practical and significant mitigation of the identity-location problem. Compared with plain TCP, QUIC can keep an ongoing connection alive across a path or address change. Compared with TCP reconnect, QUIC can avoid treating the event as a full connection failure. Its contribution is best understood as transport-level continuity under changing network attachment, rather than as a guaranteed timing improvement or complete identity-location separation.

Future work could extend the evaluation in several directions. First, the experiments could be repeated under different delay, loss, and bandwidth conditions to measure the performance cost of migration. Second, NAT rebinding and load-balanced server deployments could be evaluated more directly. Third, multipath QUIC extensions could be compared with base QUIC migration to distinguish continuation across path change from simultaneous use of multiple paths. These extensions would provide a broader view of how QUIC and related mechanisms support mobility and multihoming in realistic network environments.
